# C05 · 设计契约（D1 锁定版 · v7.3.1 · 策划/程序/美术三方对齐）

> **来源**：C01–C04 全部交付物
> **任务**：C05（4–5h）· 策划 + 程序 + 美术三方对齐会
> **状态**：✅ 草稿就位 · ⚠️ **需三方签字确认**（待 D1 下午 5h 三方会议后定稿）

---

## 一、目的

**消除三方协同中的命名歧义**。所有策划/程序/美术交付物中涉及的字段名、ID、资产名、prompt 字段名，本契约是**唯一权威**。

> **违反契约的代价**：程序用了策划未定义的字段名 → Agent 决策读取失败；美术做了未在契约中的 ID → 资源加载失败。
> **好处**：D2 mock / D3 LLM / D2 美术出图 / D3 集成，全程零沟通成本。

## 四、Prompt 字段名（策划 ↔ 程序 ↔ LLM 三方对齐 · 11b §一 8 模块）

> 每个君主 Agent 的 system prompt 由 8 个固定模块组成，模块顺序严格。

```markdown
# 1. 角色定位
{role_identity}

# 2. 性格画像
{personality}

# 3. 核心利益
{core_interest}

# 4. 可用动作
{available_actions}

# 5. 近臣倾向
{minister_tendency}

# 6. 记忆滚动摘要
{memory_summary}

# 7. 决策规则
{decision_rules}

# 8. 输出 schema
{output_schema}
```

### 4.1 动态注入字段（每轮变化）

| 字段 | 类型 | 来源 | 注入时机 |
|---|---|---|---|
| `{key_event}` | String | C08 总体剧本 | 回合开始 |
| `{key_event_implication}` | String | 07 主线规则生成 | 回合开始 |
| `{memory_summary}` | String (≤ 200 字) | 04 仲裁器生成 | 每轮博弈完成 |
| `{player_intervention}` | String (null/有) | 玩家打牌后写入 | 玩家打牌时 |

### 4.2 输出 schema 字段名（与 §2.3 Action 枚举对应）

```json
{
  "round": "1|2",
  "target_country": "zhao|qi",  // 秦王只能选赵/齐，赵王只能选秦/齐，齐王只能选秦/赵
  "action": "<见 §2.3 动作枚举>",
  "reason": "≤ 50 字",
  "confidence": "0-10",
  "expected_effect": "...",
  "settle_decision": "召见|决策已定|null"
}
```

## 七、待确认项 ⚠️

> **D1 下午 5h 三方会议需要明确的事项**：

1. **类名命名风格**：本契约使用 `PascalCase`（Godot 习惯）+ `snake_case` 字段名。程序是否一致？
2. **枚举值中文 vs 英文**：本契约 Agent 动作使用**中文**（如 `"遣使离间"`），但牌/立场使用**英文**（如 `PERSUADE`、`HEZONG`）。策划与程序应统一策略——建议 Agent 动作也用英文（`ALIENATE`）便于国际化，**需会议决定**。
3. **资源 ID 命名**：本契约使用 `snake_case`（`portrait_qin_king`）。美术工具链（如 Aseprite）是否兼容？
4. **8 模块 Prompt 字段的模板位置**：策划 C16（D3）需确定是写在 `data/prompts/` 还是 `scenes/agent/`，**程序主导**。
5. **C19（D3 4–5h）补全近臣台词片段**（每近臣 5 条）使用哪个字段名注入？建议 `minister_quotes: Array[String]`，**需会议决定**。
